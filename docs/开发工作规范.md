└─────────────────────────────────────────────────────────────┘

### JSON数据结构规范

> **方案A**：JSON只存储视频路径和选择，适用于真人视频互动游戏

```json
{
  "node_id": "ch1_sc_001",
  "scene": "SC-001 神秘房间",
  "description": "场景描述（供拍摄参考）",
  "video": {
    "path": "res://assets/videos/ch1/sc001_main.ogv",
    "duration": 480
  },
  "choices": [
    {
      "id": "choice_001",
      "text": "选择文字",
      "conditions": [],
      "effects": [],
      "next_node": "ch1_sc_002a"
    }
  ]
}
```

**字段说明**：
| 字段 | 必填 | 说明 |
|------|------|------|
| `video.path` | 是 | 视频文件路径（相对于res://） |
| `video.duration` | 是 | 视频时长（秒），用于进度显示 |
| `description` | 否 | 场景描述，供拍摄团队参考 |
| `subtitles` | 否 | 可选，辅助字幕（非必需） |

### 文档职责

| 文档类型 | 位置 | 职责 | 更新者 |
|----------|------|------|--------|
| 剧情大纲 | `02_design/03_首章剧情大纲.md` | 剧情、场景、对话设计 | 剧本策划 |
| 分支流程图 | `02_design/04_分支流程图设计.md` | 分支逻辑、好感度数值 | 游戏设计师 |
| JSON数据 | `data/stories/chapter_XX.json` | 程序运行的剧情数据 | 程序员 |

### 同步检查清单

在提交剧情相关修改前，请确认：

- [ ] 设计文档已先更新
- [ ] JSON数据与设计文档内容一致
- [ ] 节点数量、选择数量匹配
- [ ] 好感度数值匹配
- [ ] 结局触发条件匹配
- [ ] 更新日志已记录

### 违规处理

如果发现JSON数据与设计文档不一致：
1. **不要直接修改JSON**
2. 先更新设计文档
3. 再同步到JSON数据
4. 在更新日志中说明同步操作

## 1. 代码规范

### 1.1 命名规范

**文件命名**
- 脚本文件：使用蛇形命名法，如 `game_manager.gd`
- 场景文件：使用蛇形命名法，如 `main_menu.tscn`
- 资源文件：使用蛇形命名法，如 `main_menu_bgm.ogg`

**变量命名**
- 局部变量：蛇形命名法，如 `current_node`
- 成员变量：蛇形命名法，下划线前缀，如 `_is_playing`
- 常量：全大写，下划线分隔，如 `MAX_CACHE_SIZE`

**函数命名**
- 使用蛇形命名法，如 `get_current_node()`
- 私有方法：下划线前缀，如 `_load_chapter_data()`
- 信号回调：使用描述性名称，如 `_on_node_changed()`

### 1.2 注释规范

**文件头注释**
```gdscript
## 管理器说明
## 作者: xxx
## 日期: YYYY-MM-DD
## 修改记录: [日期] [修改内容]
```

**函数注释**
- 公有方法：使用 `## 方法名` 格式
- 私有方法：使用 `## 私有方法名` 格式
- 复杂逻辑：添加行内注释解释

### 1.3 组织规范

使用 `#region` 和 `#endregion` 分组代码：
```gdscript
#region 变量管理
func set_variable(...) -> void:
    ...
#endregion

#region 音频播放
func play_bgm(...) -> void:
    ...
#endregion
```

### 1.4 信号规范

- 信号命名使用过去式或名词形式
- 在信号下方添加注释说明用途
```gdscript
signal node_changed(node_id: String, node_data: Dictionary)
## 节点变更时发出
```

## 2. Git 工作流

### 2.1 分支策略

```
main        # 生产分支，始终保持稳定
develop     # 开发主分支
feature/*   # 功能开发分支（如 feature/save-system）
fix/*       # Bug 修复分支（如 fix/video-loading）
hotfix/*    # 紧急修复分支
```

### 2.2 提交规范

**提交信息格式**：
```
<type>(<scope>): <subject>

<body>

<footer>
```

**Type 类型**：
- `feat`: 新功能
- `fix`: Bug 修复
- `docs`: 文档更新
- `refactor`: 代码重构
- `test`: 测试相关
- `chore`: 构建/工具链相关

**示例**：
```
feat(story): 添加分支选择条件判断系统

修复了选择按钮显示问题，现在会根据玩家状态自动锁定/解锁选项。
