## 一、Git分支管理规范

### 1.1 分支策略

```
main (主分支 - 稳定发布版本)
  │
  ├── develop (开发分支 - 最新开发版本)
  │     │
  │     ├── feature/story-engine (功能分支)
  │     ├── feature/video-player (功能分支)
  │     ├── feature/save-system (功能分支)
  │     │
  │     ├── bugfix/video-loading (修复分支)
  │     │
  │     └── refactor/ui-components (重构分支)
  │
  └── release/v1.0.0 (发布分支)
```

### 1.2 分支命名规范

| 分支类型 | 命名格式 | 示例 | 说明 |
|----------|----------|------|------|
| 功能分支 | `feature/功能名称` | `feature/story-engine` | 新功能开发 |
| 修复分支 | `bugfix/问题描述` | `bugfix/video-loading` | Bug修复 |
| 重构分支 | `refactor/重构内容` | `refactor/ui-components` | 代码重构 |
| 发布分支 | `release/版本号` | `release/v1.0.0` | 版本发布 |
| 热修复分支 | `hotfix/问题描述` | `hotfix/critical-crash` | 线上紧急修复 |

### 1.3 分支操作规范

```
1. 从develop分支创建feature分支
   git checkout develop
   git pull origin develop
   git checkout -b feature/story-engine

2. 开发完成后提交PR
   git add .
   git commit -m "feat(story): 实现剧情引擎核心逻辑"
   git push origin feature/story-engine

3. 创建Pull Request，等待Code Review

4. Review通过后合并到develop

5. 删除已合并的feature分支
```

### 1.4 分支保护规则

| 分支 | 保护规则 |
|------|----------|
| `main` | 禁止直接推送，必须通过PR合并，需要1个Review通过 |
| `develop` | 禁止直接推送，必须通过PR合并，需要1个Review通过 |
| `release/*` | 禁止直接推送，只允许修复Bug |

## 二、代码提交规范

### 2.1 Commit Message格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### 2.2 Type类型

| Type | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat(story): 添加分支选择系统` |
| `fix` | Bug修复 | `fix(video): 修复视频加载卡顿问题` |
| `docs` | 文档更新 | `docs(readme): 更新安装说明` |
| `style` | 代码格式 | `style(ui): 调整按钮样式` |
| `refactor` | 重构 | `refactor(save): 重构存档系统` |
| `test` | 测试 | `test(story): 添加剧情引擎单元测试` |
| `chore` | 构建/工具 | `chore: 更新Godot版本配置` |

### 2.3 Scope范围

| Scope | 说明 |
|-------|------|
| `story` | 剧情系统 |
| `video` | 视频播放 |
| `save` | 存档系统 |
| `ui` | 用户界面 |
| `audio` | 音频系统 |
| `core` | 核心模块 |
| `data` | 数据处理 |

### 2.4 提交示例

```bash
# 功能开发
git commit -m "feat(story): 实现剧情节点跳转功能"

# Bug修复
git commit -m "fix(video): 修复Safari浏览器视频播放兼容问题

- 添加WebM格式支持
- 优化视频预加载策略

Closes #123"

# 文档更新
git commit -m "docs(api): 更新存档系统API文档"
```

```gdscript
    var target_quality = quality if quality else DEFAULT_QUALITY
    _do_load(video_id, target_quality)

## 私有方法
func _do_load(video_id: String, quality: String) -> void:
    # 实现细节
    pass
```

### 3.2 命名约定

| 类型 | 命名风格 | 示例 |
|------|----------|------|
| 文件名 | snake_case | `video_manager.gd` |
| 类名 | PascalCase | `VideoManager` |
| 函数名 | snake_case | `load_video()` |
| 私有函数 | _snake_case | `_do_load()` |
| 变量名 | snake_case | `video_id` |
| 私有变量 | _snake_case | `_video_pool` |
| 常量 | UPPER_SNAKE_CASE | `MAX_CACHE_SIZE` |


## 五、任务管理规范

### 5.1 任务看板结构

```
📋 待办 (To Do)
  │
  ├── 🔄 进行中 (In Progress)
  │     │
  │     └── 👀 评审中 (Review)
  │           │
  │           └── ✅ 完成 (Done)
  │
  └── 🚫 阻塞 (Blocked)
```

### 5.2 任务卡片格式

```markdown
## 任务标题

**类型**: 功能/Bug/优化/文档
**优先级**: P0(紧急)/P1(高)/P2(中)/P3(低)
**负责人**: XXX
**截止日期**: YYYY-MM-DD
**预估工时**: X小时

### 描述
任务详细描述...

### 验收标准
- [ ] 标准1
- [ ] 标准2

### 关联
- 关联任务: #XXX
- 关联文档: [文档名](链接)
```

### 5.3 任务状态流转

```
待办 → 进行中 → 评审中 → 完成
         ↓
       阻塞 → 进行中（问题解决后）
```

### 5.4 任务优先级定义

| 优先级 | 定义 | 响应时间 |
|--------|------|----------|
| P0 | 紧急，影响发布或阻塞他人 | 立即处理 |
| P1 | 高优先级，本周必须完成 | 24小时内响应 |
| P2 | 中优先级，本月需完成 | 3天内响应 |
| P3 | 低优先级，有空时处理 | 一周内响应 |

## 六、Code Review规范

### 6.1 Review流程

```
1. 开发者完成开发，创建PR
2. 填写PR模板（变更说明、测试情况）
3. 指定Reviewer（至少1人）
4. Reviewer进行代码审查
5. 提出修改意见或批准
6. 开发者修改或合并
```

### 6.2 Review检查项

| 检查项 | 说明 |
|--------|------|
| 功能正确性 | 代码是否实现了预期功能 |
| 代码规范 | 是否符合项目编码规范 |
| 性能影响 | 是否有性能问题 |
| 安全性 | 是否存在安全风险 |
| 测试覆盖 | 是否有足够的测试 |
| 文档更新 | 相关文档是否更新 |

### 6.3 PR模板

```markdown
## 变更说明
简要描述本次变更的内容...

## 变更类型
- [ ] 新功能
- [ ] Bug修复
- [ ] 重构
- [ ] 文档更新

## 测试情况
- [ ] 本地测试通过
- [ ] 单元测试通过
- [ ] 无回归问题

## 相关Issue
Closes #XXX

## 截图/演示
（如有UI变更，附上截图）
```

## 七、会议规范

### 7.1 会议类型

| 会议 | 频率 | 时长 | 参与人 | 形式 |
|------|------|------|--------|------|
| 每日站会 | 每日 | 15分钟 | 核心团队 | 站立/线上 |
| 周例会 | 每周五 | 1小时 | 全员 | 正式会议 |
| 里程碑评审 | 按计划 | 2小时 | 全员+发起人 | 正式评审 |
| 技术评审 | 按需 | 1小时 | 技术团队 | 技术讨论 |

### 7.2 每日站会内容

```
每人发言1-2分钟，回答三个问题：
1. 昨天完成了什么？
2. 今天计划做什么？
3. 有什么阻塞或需要帮助？
```

### 7.3 周例会内容

```
1. 本周进度回顾（15分钟）
2. 下周计划（10分钟）
3. 风险与问题（15分钟）
4. 其他事项（10分钟）
5. Q&A（10分钟）
```

### 7.4 会议纪要要求

- 所有正式会议需要记录纪要
- 纪要包含：决议、行动项、责任人、截止日期
- 纪要发布时间：会议后24小时内
- 纪要归档位置：`docs/sprints/`

## 八、居家办公协作规范

### 8.1 工作时间要求

| 项目 | 要求 |
|------|------|
| 核心工作时间 | 10:00 - 18:00（必须在线） |
| 每日站会时间 | 09:30（线上） |
| 响应时间 | 工作时间内消息30分钟内回复 |
| 离线报备 | 离开超过1小时需在企业微信群报备 |

### 8.2 远程协作工具

| 用途 | 工具 | 说明 |
|------|------|------|
| 即时通讯 | 企业微信 | 日常沟通、通知公告 |
| 视频会议 | 腾讯会议 | 站会、周会、评审 |
| 文档协作 | Notion/Git | 项目管理、文档维护 |
| 代码协作 | GitHub | 代码评审、Issue跟踪 |
| 设计协作 | Figma | UI设计、评审 |
| 文件共享 | 企业微信/云盘 | 大文件传输 |

### 8.3 远程会议规范

| 会议类型 | 要求 |
|----------|------|
| 每日站会 | 提前5分钟进入会议室，开启摄像头 |
| 周例会 | 提前10分钟进入会议室，开启摄像头 |
| 评审会议 | 提前准备好演示材料，共享屏幕 |
| 临时会议 | 通过企业微信群发起，注明议题和时间 |

### 8.4 远程沟通礼仪

1. **消息发送**
   - 使用@提醒相关人员
   - 紧急事项同时电话通知
   - 非工作时间消息不要求即时回复

2. **文件共享**
   - 使用云盘链接而非直接发送大文件
   - 文件命名规范：`日期_项目_内容_版本.扩展名`

3. **屏幕共享**
   - 提前关闭无关窗口和通知
   - 演示时开启高清模式
   - 需要讨论时共享编辑权限

### 8.5 设备与网络要求

| 项目 | 最低要求 |
|------|----------|
| 电脑 | 可运行Godot 4.x，内存≥16GB |
| 网络 | 稳定宽带，上传速度≥10Mbps |
| 摄像头 | 720p及以上 |
| 麦克风 | 清晰可辨，建议使用降噪麦克风 |
| 显示器 | 建议2台显示器提高效率 |

## 九、紧急问题处理

### 8.1 问题分级

| 级别 | 定义 | 响应时间 | 处理人 |
|------|------|----------|--------|
| L1 | 严重影响，系统不可用 | 15分钟 | 技术负责人+主程 |
| L2 | 功能异常，影响使用 | 1小时 | 相关开发人员 |
| L3 | 轻微问题，不影响使用 | 4小时 | 相关开发人员 |
| L4 | 优化建议 | 24小时 | 排期处理 |

### 8.2 紧急问题处理流程

```
1. 发现问题 → 企业微信群报告
2. 评估级别 → 确定响应人
3. 问题定位 → 快速分析原因
4. 临时修复 → 先恢复服务
5. 根本修复 → 彻底解决问题
6. 复盘总结 → 防止再次发生
```
